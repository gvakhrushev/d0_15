# BOOK 05 — Verification, status and certificate discipline

## 05.1 Standard-language audit rule

A claim is not release-ready merely because a D0-local word appears in the text. The claim must expose its standard object:

```text
archive -> traced-out complement or environment;
forgetting -> conditional expectation / partial trace / RG map;
readout -> positive operator-valued instrument;
bridge -> coupling, functor, or EFT matching map;
passport -> external comparison protocol;
stiffness -> structural uniqueness plus negative controls.
```

This is now part of the proof-cell audit. Undefined D0-local terminology is a release blocker, not a stylistic issue.

## 05.2 . Role of Book 05

Book 05 is the audit calculus of the D0 corpus. It does not introduce the condensed site, the detector algebra, the action stack or matter spectra. Those belong to Books 01--04. Book 05 defines when a proposed D0 statement is a theorem, a finite readout, a passport, an external bridge, a diagnostic row or an open problem.

The normative verification chain is

```math
\boxed{\text{claim}\rightarrow\text{support}\rightarrow\text{probe/operator}\rightarrow R=D^\dagger D\rightarrow\text{finite quotient}\rightarrow\text{certificate/passport}\rightarrow\text{status}.}
```

In this book a certificate never replaces a proof. A certificate is an executable or reproducible witness that the finite claim stated elsewhere has been evaluated under its allowed inputs and forbidden shortcuts.

## 05.3 . Verification is itself a finite measurement

The Popper layer of D0 is not an informal demand for experimental friendliness. It is a finite-detector condition. A statement is admissible only if it can be reduced to a stable finite record, a finite operator trace, an invariant quotient or an explicitly external passport.

No-monopoly verification is inherited from Book 01:

```math
\text{no interaction without information},\qquad \text{no information without detector/register}.
```

The verification object is therefore not an opinion about a formula. It is a finite comparison between an addressed claim and a registered response.

```math
\text{verifiable claim}\neq\text{beautiful expression};\qquad \text{verifiable claim}=\text{finite record with a falsification hook}.
```

## 05.4 . Proof, certificate and passport separation

D0 uses three different evidential levels.

1. **Proof.** A mathematical statement derived from the D0 condensed/profinite support, finite operators, invariant quotients or sector constructions.
2. **Certificate.** A finite computation or symbolic check that confirms a specified theorem, formula, uniqueness condition, no-go boundary or internal arithmetic.
3. **Passport.** A comparison with an external physical, mathematical or computational convention after the D0 object has already been fixed.

The separation rule is

```math
\boxed{\text{external data may validate a passport but may not define the D0 object.}
```

The strongest version of this rule is the anti-numerology firewall:

```math
\text{closed internal object}\rightarrow\text{declared bridge}\rightarrow\text{external comparison}
```

not

```math
\text{external table}\rightarrow\text{search for a D0-looking expression}
```

## 05.5 . Claim object schema

Every mature D0 claim must be represented by the following fields:

| Field | Meaning |
|---|---|
| `Claim_ID` | stable identifier |
| `Source_Books` | canonical book locations |
| `Support` | condensed/profinite, graph, operator or sector support |
| `Probe_or_Gate` | finite test object or action gate |
| `Linear_Exchange_or_Operator` | operator before quadratic response |
| `Quadratic_Readout` | response, trace, spectrum, quotient or finite probability |
| `Runtime_or_Window` | tick, heat, S_DE or transfer window when needed |
| `Bridge_or_Dictionary` | external comparison map if any |
| `Certificate_Scripts` | executable or manual witness set |
| `External_Inputs_Allowed` | what may be imported |
| `Forbidden_Shortcuts` | what would invalidate the claim |
| `Falsification_Hook` | how the claim can fail |
| `Next_Update_Needed` | exact maturation step |

The generic claim object can be written as

```math
\tau=(S,\rho,D,R,Q,\beta,\nu,\theta),
```

where S is support, D is the finite detector/operator channel, R is the positive response, Q is the quotient or passport target, beta records bridge inputs, nu records forbidden shortcuts, and theta records the falsification hook.

## 05.7 . Promotion ladder

A claim is promoted only by climbing the ladder in order:

```math
\text{definition}\rightarrow\text{internal theorem}\rightarrow\text{certificate}\rightarrow\text{passport}\rightarrow\text{external bridge or empirical pass}.
```

The ladder has four hard stops.

1. **Support stop.** If only a support invariant is known, the claim may not be read as a mass or empirical coefficient.
2. **Operator stop.** If the finite operator is not fixed before comparison, the claim is open.
3. **Bridge stop.** If an external convention enters, the bridge must be declared.
4. **Falsification stop.** If no failure condition is stated, the claim remains diagnostic.

A certificate can promote only the part it tests. For example, an exact Hodge spectrum certificate promotes a support theorem, not a full hadron mass table.

## 05.8 . Forbidden shortcuts

The following shortcuts are invalid across the corpus:

```math
3960\not\rightarrow m_p,\qquad 400\not\rightarrow m_\pi
```

The first line forbids reading the baryon Hodge support gap directly as the proton mass, and the second forbids reading the meson lower-Hodge seed directly as a pion mass.

Other forbidden shortcuts:

- using M_Z, M_W, m_p, H_0, G_N or a PDG table as a hidden D0 anchor;
- choosing CKM phases or permutations after looking at the external table;
- moving BAO shape parameters after comparison with survey data;
- importing a mainstream theorem as if it were a D0 derivation;
- treating a no-go result as a temporary inconvenience to be patched by a free parameter.

The legal direction is always

```math
\text{D0 construction first},\qquad \text{external dictionary second},\qquad \text{comparison third}.
```

## 05.9 . Negative controls and no-go results

Negative controls are active evidence. They protect the theory from numerological overreach.

Important no-go classes include:

```math
\text{support invariant without transfer}\not\rightarrow\text{physical mass}
```

```math
\text{coefficient closure without runtime window}\not\rightarrow\text{dynamic observable}
```

```math
\text{terminal readout analogy}\not\rightarrow\text{claim that all interactions are EM}
```

A no-go result receives theorem-boundary status when it is derived from the current operator set. The baryon multiplet result is an example: the nucleon line is integrated, while Delta, Lambda and Omega require an invariant spin/flavour/decuplet operator before promotion.

## 05.10 . Certificate tiers

The certificate surface is deliberately smaller than the historical script surface.

| Tier | Role | Public meaning |
|---|---|---|
| Tier 1 | canonical core cert | safe internal theorem witness |
| Tier 2 | empirical/passport cert | external data or literature bridge required |
| Tier 3 | workbench/helper cert | development or sector exploration |
| Tier 4 | retired/mutating/generator | scaffold, not public evidence |

A script that mutates files, generates books or depends on missing data is not a Tier-1 theorem certificate. It may still be valuable, but it must not sit on the public proof path.

The canonical certificate row has the form

```math
c=(id,\tau,inputs,hash,result,date,scope,mutation,externality).
```

## 05.11 . External-data discipline

External data are allowed only in declared passports. The bridge must specify the imported convention, the frozen D0 object and the comparison metric.

For an external observable O_ext, the legal comparison has the form

```math
O_{D0}\text{ fixed},\qquad B:O_{D0}\to O_{ext},\qquad \text{residual}=d(B(O_{D0}),O_{ext}).
```

The illegal comparison is

```math
O_{ext}\to\text{choose }O_{D0}\text{ after the fact}.
```

For data-heavy passports such as BAO/S_DE, Book 05 requires a data manifest, a hash ledger or explicit EXTERNAL-DATA-REQUIRED status. A textual PASS without reproducible inputs is not a complete external audit pass.

## 05.12 . Sector verification protocols

### 05.12.1 Foundations

Foundational claims must be checked by finite algebra, condensed/profinite admissibility, response positivity or quotient uniqueness. Examples: no-monopoly verification, minimal record split, finite runtime and Born/Hilbert trace ratio.

### 05.12.2 Action and vertices

Action claims must specify the gate, scene functional, single action section and boundary/curvature response. Runtime and witness transport must remain finite.

### 05.12.3 Matter and hadrons

Matter claims are typed as support, readout or passport. Hodge spectra and Schur complements are not masses unless a sector transfer is supplied. Charged-lepton, CKM, proton/neutron and photon-scattering claims use explicit finite operators and are verified under their own scope.

The status book does not define theorem-specific demotion clauses. Numerical rows are audited by the uniform proof-cell completeness rule below and by the architectural proof-cell ledger.

### 05.12.4 Electroweak sector

EW claims must distinguish radial-depth closure, pole comparisons, Higgs anchor conventions and loop/radiative dressing. A bare internal ratio is not the same object as a fully dressed electroweak fit.

### 05.12.5 CKM sector

CKM verification has three stages:

```math
\text{internal orientation}\rightarrow\text{mass-basis dressing}\rightarrow\text{external convention comparison}.
```

No row/column or phase convention may be chosen after comparison.

### 05.12.6 Gravity and cosmology

Gravity and cosmology claims must distinguish single-section outputs, archive-pressure operators, survey-driver amplitudes and S_DE dynamic transfer. BAO/S_DE is an empirical passport only if data dependencies and fixed pre-comparison parameters are declared.

## 05.15 . Reproducibility requirements

Each public release must include:

1. theorem/status database;
2. certificate-to-claim map;
3. primary cert per active claim where available;
4. external data manifest;
5. reproducibility ledger with PASS, FAIL, MISSING_DATA, MISSING_MODULE, MANUAL_ONLY, STALE or EXTERNAL-DATA-REQUIRED;
6. retired-scaffold manifest.

The important release invariant is

```math
\text{claim coverage}\neq\text{script count}.
```

A corpus with fewer scripts and better claim coverage is stronger than a corpus with hundreds of unclassified files.

## 05.16 . What Book 05 proves

Book 05 proves the verification grammar of the corpus:

```math
\boxed{\text{D0 claim status is typed by construction, certificate, bridge and falsification hook.}
```

It does not prove every sector formula. Instead it prevents sector formulas from being overstated, under-typed or promoted by analogy.

## 05.17 . Recovered source-layer import discipline

Certified theorem notes are admissible only after their proof content is ported into the active narrative through a stricter import rule.

A historical theorem note may enter the active corpus only as one of:

1. a core theorem block;
2. a guardrail or no-go block;
3. a passport/certificate dependency;
4. a retired diagnostic superseded by a later theorem.

No stale diagnostic may override a later closure.  In particular, the following older open labels are superseded unless a new contradiction is exhibited:

```text
EW absolute radial/Higgs scale as a second GeV anchor;
CKM missing registered operator;
collider runtime as event-count oracle;
N-core disconnected from proton/neutron readout;
B-terminal requiring a dark-particle shortcut.
```

The active status rule is:

```text
historical diagnostics are retained only if marked as superseded, retired, or external application contracts.
```

This recovered rule protects the new books from re-importing old workbench uncertainty as if it were an active open core theorem.

## 05.18 General architectural proof-cell completeness rule

A D0 claim is not promoted by the presence of a result, a numerical value, a claim ID, or a certificate name alone. Promotion requires a complete proof cell. The proof cell is a structured object:

```text
Claim ID
→ theorem statement
→ support object
→ probe/gate
→ finite linear exchange or operator
→ quadratic response/readout
→ runtime or finite window, if the claim is dynamic
→ quotient/resolution or terminal section
→ bridge/dictionary, if the claim touches external physics
→ certificate/passport or explicit no-certificate reason
→ forbidden shortcuts
→ falsification hook
```

This rule is claim-independent. It is not a special exception for any numerical row. If an active book states an output without its proof cell, the status is automatically lowered to `PROOF-CELL-MISSING` for the release path, even if an older archive or a script contains the missing derivation. If the derivation exists only in a script, the book must still expose the mathematical cell at the level needed for hostile review.

A book passes proof-cell review only if its promoted claims expose the following items in the active narrative, not merely in a CSV row:

```text
support object
probe/gate or admissible morphism
finite linear exchange/operator
quadratic response or spectral functional
runtime/window if dynamic
quotient/resolution/terminal section
bridge/dictionary if external
forbidden shortcut
falsification hook
```

This applies to all claims: foundational, matter, gravity, cosmology, carrier and certificate-orchestration rows.

For numerical rows the minimum proof cell additionally requires the dimensional section, the source finite operator or trace, the exact arithmetic identity used, and the anti-anchor condition. A naked number is not a theorem.

The canonical audit artifact for this rule is:

```text
03_PROOF_CELL_LEDGER/D0_ARCHITECTURAL_PROOF_CELL_AUDIT.csv
```

## 05.19 . Bridge-claim obligations for QFT/EFT, RG and scheme-dependent passports

A claim that compares a D0 finite invariant with a QFT, EFT, PDG or collider convention is not complete merely because the internal invariant is complete.  It must expose a bridge proof cell.

The bridge proof cell has the following additional fields:

```text
bare D0 object;
external theory or EFT used;
matching scale;
renormalization scheme;
field content;
beta functions / anomalous dimensions / threshold conventions;
Ward/anomaly preservation condition;
forbidden new parameters;
external observable or table;
scheme-change rule;
falsification hook.
```

If these fields are absent, the claim may keep its internal D0 status but its external comparison status is downgraded to `BRIDGE-PROOF-CELL-MISSING`.

### 05.19.1 Scheme discipline

D0 core invariants are scheme-independent because they are finite internal objects.  A QFT-dressed comparison is scheme-indexed:

```math
O_{\mathrm{dressed}}
=
\mathfrak D_{\mathrm{SM}}^{D0}(\mathsf S,\mu,\Lambda_{\mathrm{act}})
[O_{\mathrm{bare}}].
```

The scheme `\mathsf S` and scale `\mu` are part of the claim.  Changing them after seeing a comparison table is a forbidden shortcut.

### 05.19.2 Ward and anomaly discipline

For bridge claims involving gauge theory, the bridge must say how gauge consistency is preserved.  The minimal admissible answer is not "finite operators are unitary"; it is:

```text
finite carrier/dictionary fixed;
anomaly-admissible SM representation fixed;
gauge-covariant SM/EFT running used;
no new field content or fitted counterterm introduced;
comparison scheme declared.
```

This makes Ward/Slavnov--Taylor preservation an external-bridge condition, not a vague metaphysical property of the finite graph.

### 05.19.3 Constructive proof-assistant discipline

A promoted core D0 theorem should be reducible to a constructive finite or profinite proof object.  A certificate should be runnable as a finite computation.  A passport may use floating-point external comparison, but the floating-point layer is never the proof.

For future formalization, the expected mapping is:

```text
Book theorem        → proof-assistant theorem / type;
core certificate    → finite executable evaluation;
empirical passport  → external comparison script with manifest;
bridge passport     → declared external theory, scheme and data convention.
```

A claim that cannot be stated in this hierarchy must be retyped before promotion.

## 05.21 Release ledger for hostile review

The release path is now governed by a single hostile-audit matrix rather than by theorem-specific exceptions.  Each claim row is checked against:

```text
proof-cell completeness
certificate availability
actual active run status
external-data requirement
QFT/EFT bridge completeness, when applicable
```

The release authority is the theorem/status database plus the current proof-cell and certificate ledgers.
```

Status names in the books are scientific claims; release declarations are audit claims.  When they disagree, the release ledger is stricter.  In particular:

```text
EMPIRICAL-PASSPORT + missing data manifest/run status
→ external-data-required passport, not archive-reproducible pass.

ACTIVE-PASSPORT or BARE-D0-READOUT + missing bridge cell
→ bridge review, not public empirical closure.

CORE claim + complete internal proof cell but no external certificate
→ core-internal theorem may remain, but no empirical/passport claim is added.
```

### 05.21.1 Primitive-equation proof-cell requirement

The equation

```math
p+p^2=1
```

must not be treated as a naked scalar postulate in promoted foundation claims. A valid proof-cell must expose:

1. the finite-stage detector support;
2. the non-monopoly direct/return dyad;
3. constructive composition of branch weights;
4. the positive-response reason for the quadratic return;
5. single-halt normalization;
6. uniqueness of the positive solution;
7. exclusion of linear, higher-power, fractional-power and free-return alternatives.

The primitive assumptions are the finite detector obligations, not the scalar equation. If a text says only "choose `p+p^2=1`" or "nature selects `\varphi`" without the positive-response closure proof, the associated foundation row is demoted to `PRIMITIVE-EQUATION-PROOF-MISSING`.

## 05.22 . Substantive closure-cell rule for promoted claims

The release corpus no longer accepts audit-only closure.  For every promoted claim the active book must contain a closure cell with:

```text
support object → operator/exchange → quadratic or trace readout
→ quotient/window/terminal section → bridge boundary → cert artifacts → falsification hook.
```

A certificate map, status row or claim ID is not enough.  If the proof-cell content lives only in an old result file, it must be ported into the relevant active book or the claim must be demoted. ## 05.24 Release-status theorem

A status is a theorem about the documentary and mathematical state of a claim.  It is not a label of confidence.

A claim is `CORE-CLOSED` only if the active books expose a support object, finite operator, positive response or trace, quotient/window and falsification hook.  A claim is `BRIDGE-CLOSED` only if the external dictionary is typed.  A claim is `EMPIRICAL-PASSPORT` only if external data, scheme and run order are manifested.  A claim is `EXTERNAL-DATA-REQUIRED` when the internal theory is closed but reproduction requires files not bundled in the release.  A claim is `NO-GO` when the current operator set cannot generate the sector without adding an unproven operator.

This release-status theorem is applied uniformly to coefficients, matter, gravity, cosmology and detector foundation claims.
